
ТЕХНИЧЕСКОЕ ЗАДАНИЕ
Хакатон 2025
MedPartners
Автоматическая обработка архива прайсов клиник-партнёров и формирование базы услуг и цен

1. Цель и контекст
Создать систему автоматической обработки архива прайс-листов клиник-партнёров, которая:

	•	Разбирает входящие документы различных форматов (PDF, DOCX, XLSX, сканы)
	•	Извлекает структурированные данные: клиника, услуга, цена (с учётом тарифов резидент/нерезидент)
	•	Нормализует данные, приводя их к единому целевому справочнику услуг
	•	Формирует верифицированную базу данных партнёров, услуг и цен
	•	Предоставляет API и интерфейс для поиска: кто оказывает услугу и по какой цене

Целевой справочник медицинских услуг будет предоставлен организаторами в момент старта хакатона. Команды должны предусмотреть механизм загрузки справочника и привязки к нему всех извлечённых услуг.
2. Входные данные
2.1 Архив прайс-листов
Организаторы предоставляют архив (ZIP) с прайсами клиник-партнёров. Архив содержит документы следующих типов:

Один файл = прайс одной клиники-партнёра на конкретную дату. Имя файла может содержать название клиники и/или дату.Для одного партнера может быть несколько прайсов, брать все доступные и выстраивать историю изменения цен.

4. Функциональные требования
4.1 Загрузка и первичная обработка архива
	•	Приём архива ZIP через интерфейс или CLI-команду
	•	Автоматическое определение типа каждого файла
	•	Постановка файлов в очередь обработки с сохранением статуса
	•	Сохранение оригинальных файлов для возможности повторной обработки
4.2 Извлечение данных по формату
PDF (текстовый):
	•	Извлечение текста и таблиц (pdfplumber, PyMuPDF или аналог)
	•	Распознавание структуры прайса: заголовок таблицы, строки, цены

PDF (скан):
	•	OCR с помощью Tesseract OCR или облачного API (Google Vision, AWS Textract и др.)
	•	Постобработка: очистка артефактов OCR перед извлечением данных

XLSX / XLS:
	•	Обход всех листов книги
	•	Определение строки с заголовками (может быть не первой строкой)
	•	Извлечение колонок с названием услуги и ценой/ценами

DOCX (включая tracked changes):
	•	Принятие всех отслеживаемых изменений — работа с финальной версией текста
	•	Извлечение таблиц и их структурирование
4.3 Нормализация и сопоставление со справочником
	•	Автоматическое сопоставление service_name_raw с записями справочника по точному совпадению, синонимам и нечёткому поиску
	•	Порог уверенности сопоставления: конфигурируется (например, cosine similarity > 0.85 → автосопоставление, иначе → очередь на ревью)
	•	Несопоставленные позиции выделяются в отдельную очередь unmatched для ручной разметки
	•	Интерфейс ручного сопоставления: оператор выбирает из справочника или создаёт новую позицию
4.4 Валидация и верификация данных
Автоматические проверки (выполняются при парсинге):

Проверка
Действие при нарушении
Цена > 0 и является числом
Запись в лог, статус needs_review
Цена нерезидента >= цены резидента
Предупреждение, флаг для ревью
Название услуги не пустое
Пропуск строки, запись в лог
Дата прайса не в будущем
Предупреждение
Дублирование позиции (та же клиника, услуга, дата)
Деdup: сохранить новую, архивировать старую
Цена отличается от предыдущей версии > 50%
Флаг аномалии, требует ручного подтверждения
Валюта не KZT
Конвертировать по курсу на дату прайса, сохранить оригинал
Документ не содержит распознаваемых данных
Статус error, уведомление оператора

Ручная верификация:
	•	Интерфейс очереди верификации с отображением: исходный документ (или его фрагмент), извлечённые данные, предложенное сопоставление
	•	Оператор подтверждает, отклоняет или корректирует каждую позицию
	•	Версионирование: при изменении цены старая версия архивируется, не удаляется

4.6 Интерфейс пользователя
	•	Поиск услуги → список партнёров с ценами резидент/нерезидент
	•	Страница партнёра: полный прайс, контакты, дата актуальности
	•	Административный раздел: загрузка архива, статус обработки, очередь верификации
	•	Дашборд: количество обработанных документов, процент успешной нормализации, позиции в очереди

7. Результаты, которые сдаёт команда
	•	Рабочий MVP: репозиторий с инструкцией запуска (README)
	•	Обработанная база данных: результаты разбора предоставленного архива
	•	Сформированный справочник услуг (если не был предоставлен организаторами в финальном виде)
	•	Документация API (OpenAPI / Swagger)
	•	Отчёт о качестве обработки: количество документов, % успешной нормализации, количество позиций в очереди
	•	Презентация решения: 5–7 слайдов (архитектура, демо, метрики качества)
	•	Демо-видео (опционально, 1–3 минуты)
