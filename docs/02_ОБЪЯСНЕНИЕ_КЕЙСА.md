# MedPartners — объяснение Кейса 2 простым языком

> Для PM и команды: что мы строим, зачем и как это выглядит снаружи

---

## 1. Проблема, которую мы решаем

Медицинская сеть (заказчик хакатона) работает с **8 клиниками-партнёрами**. У каждой клиники свой прайс-лист — и все они разные:

- Кто-то присылает **Excel**, кто-то **Word**, кто-то **PDF на 85 страниц**
- Названия услуг не совпадают: в одной клинике «ОАК», в другой «Общий анализ крови с лейкоцитарной формулой»
- Цены разные для **резидентов** и **нерезидентов**
- Прайсы обновляются каждый год — нужна **история изменений**

Сейчас всё это разбирают **вручную**. Наша задача — автоматизировать.

---

## 2. Что мы строим

**MedPartners** — система, которая:

```
Прайсы клиник (PDF, Excel, Word)
            ↓
    Автоматический разбор
            ↓
  Сопоставление со справочником (1286 услуг)
            ↓
    База данных: кто, что, по какой цене
            ↓
  Поиск: «Сколько стоит УЗИ?» → таблица цен по всем клиникам
```

Представь **Aviasales, но для медицинских услуг** — только данные берутся не с сайтов, а из **внутреннего архива прайсов партнёров**.

---

## 3. Кто пользуется системой

### Оператор (админка)
- Загружает новый прайс (ZIP или файл)
- Видит статус обработки: «готово», «ошибка», «нужна проверка»
- Разбирает очередь **unmatched** — услуги, которые система не смогла сопоставить автоматически
- Подтверждает или исправляет цены

### Менеджер / внутренний пользователь (поиск)
- Ищет услугу: «Приём терапевта»
- Видит: Клиника 1 — 14 400 ₸, Клиника 7 — 12 600 ₸, Клиника 2 — 10 800 ₸
- Фильтрует по городу
- Открывает карточку клиники с контактами

## 4. Наши данные

### Справочник услуг (`Справочник услуг.xlsx`)
Это **эталон** — 1286 медицинских услуг с официальными названиями и кодами тарификатора. Все прайсы клиник должны быть **привязаны** к этому справочнику.

Пример:

| Специальность | Название | Код тарификатора |
|---|---|---|
| ИФА | Исследование на ВИЧ | B03.016.003 |
| УЗИ | УЗИ органов брюшной полости | C03.015.002 |

### Прайсы клиник (`прайсы/`)

| Клиника | Файлы | Как обрабатываем |
|---|---|---|
| Клиника 1 | DOCX 2024 + PDF 2026 | Программно (таблицы читаются) |
| Клиника 2 | PDF 2025 + PDF 2026 | LLM (битый текст, резидент/нерезидент) |
| Клиника 3 | PDF 2026 | LLM (лаборатория, OCR-артефакты) |
| Клиника 4 | PDF 2026 | LLM (3 ценовых колонки) |
| Клиника 5 | PDF 2025 | LLM (нестандартная вёрстка) |
| Клиника 6 | XLSX 2026 | Программно (~5000 строк) |
| Клиника 7 | XLS 2026 | Программно (~3000 строк) |
| Клиника 8 | XLSX 2026 | Программно (~1800 строк) |

### Метаданные клиник (`partners.json`)
Реальные названия клиник в файлах не раскрыты — используем **«Клиника 1–8»**. Город, адрес, телефон — в **статическом конфиге**, который заполняем вручную. Это проще и надёжнее, чем вытаскивать контакты из PDF.

## 5. Как работает обработка (гибридный пайплайн)

### Шаг 1: Загрузка
Оператор кладёт файл (или ZIP с архивом) → система определяет формат и ставит в очередь.

### Шаг 2: Извлечение данных

**Быстрый путь (программный)** — для Excel, Word и нормальных PDF:
- Библиотеки читают таблицы напрямую
- Мгновенно, точно, бесплатно
- Клиники 1, 6, 7, 8

**Умный путь (LLM)** — для «битых» PDF:
- Gemini Flash смотрит на страницу (текст + картинка)
- Восстанавливает испорченные слова («OAK» → «ОАК»)
- Вытаскивает нерегулярные таблицы в JSON
- Клиники 2, 3, 4, 5

### Шаг 3: Сопоставление со справочником

Клиника пишет «ОАК без СОЭ», справочник говорит «Общий анализ крови (ОАК)» — это **одна услуга**.

Три уровня сопоставления:

| Уровень | Как работает | Пример |
|---|---|---|
| **Exact** | Точное совпадение после очистки текста | «узи почек» = «УЗИ почек» |
| **Fuzzy** | Нечёткий поиск (RapidFuzz) | «анализ крови общий» ≈ «Общий анализ крови» |
| **LLM** | Модель понимает медицинские аббревиатуры | «ОАК» → «Общий анализ крови (ОАК)» |

Не сопоставилось автоматически → попадает в **unmatched** → оператор разбирает вручную.

### Шаг 4: Валидация
- Цена должна быть числом > 0
- Цена нерезидента ≥ цены резидента (иначе — предупреждение)
- Резкое изменение цены > 50% → флаг аномалии
- Дубликаты → оставляем новую версию, старую архивируем

### Шаг 5: Поиск
Всё ложится в PostgreSQL → пользователь ищет услугу → видит цены по всем клиникам.

## 8. Как нас оценят (критерии жюри)

| Критерий | Вес | Что важно |
|---|---|---|
| Качество извлечения данных | **30%** | Правильно разобрали Excel, PDF, DOCX; LLM для битых файлов |
| Нормализация и сопоставление | **25%** | ≥ 70% автоматически; очередь unmatched с ручной разметкой |
| Валидация и верификация | **20%** | Проверки цен, аномалии, версионирование |
| API и поиск | **15%** | Swagger, скорость, полнота эндпоинтов |
| UX админки | **10%** | Удобство оператора, дашборд, unmatched UI |

**Наши сильные стороны:**
- Гибридный пайплайн (программный + LLM) — показывает зрелый подход
- Реальные данные всех 8 клиник
- Трёхуровневый matching — выше % автосопоставления
- Полноценный UI с RU/KZ

## 10. Глоссарий

| Термин | Значение |
|---|---|
| **Партнёр** | Клиника из нашего списка (Клиника 1–8) |
| **Прайс-документ** | Один файл = один прайс одной клиники на дату |
| **Позиция прайса** | Одна строка: услуга + цена |
| **Справочник** | Эталонный список 1286 услуг от организаторов |
| **Нормализация / Match** | Привязка «сырого» названия из прайса к услуге справочника |
| **Unmatched** | Позиции, которые система не смогла сопоставить автоматически |
| **Резидент / нерезидент** | Цена для граждан РК vs иностранцев |
| **Тарификатор** | Официальный код медицинской услуги (TarificatrCode) |
